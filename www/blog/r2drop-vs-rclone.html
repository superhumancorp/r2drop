R2Drop vs rclone for Cloudflare R2

Comparison
—
March 2, 2026

If you've spent any time on r/CloudFlare, you've seen the answer to "how do I upload to R2?" at least fifty times: just use rclone.

And look, rclone is great. I've used it for years. But "just use rclone" glosses over the ~20 minutes of config-file editing it takes to get there. I built R2Drop because I wanted something faster for quick uploads, and I figured other people did too.

rclone: the 800-pound gorilla

rclone is a CLI tool that talks to basically every cloud storage provider that exists. S3, R2, GCS, Dropbox, OneDrive, you name it. It can copy, sync, mount remote storage as a local drive, encrypt on the fly, and handle transfers of any size. It's been around since 2014 and it's battle-tested.

The setup for Cloudflare R2 looks something like this:

[myr2]
type = s3
provider = Cloudflare
access_key_id = your_access_key
secret_access_key = your_secret_key
endpoint = https://<account_id>.r2.cloudflarestorage.com
acl = private

You put that in ~/.config/rclone/rclone.conf, and then you can do:

rclone copy ./file.png myr2:my-bucket/path/

That works. But your credentials are sitting in a plain text config file. And if you have multiple R2 accounts, you're managing multiple config blocks manually. And there's no URL copying after upload, you have to construct it yourself based on your custom domain or the R2 public URL pattern.

R2Drop: the single-purpose alternative

R2Drop does one thing: upload files to Cloudflare R2 from macOS. It has a native GUI (menu bar app) and a CLI, and they share the same config and upload queue.

Setup takes about two minutes. You paste your Cloudflare API token, pick a bucket, optionally set a custom domain, and you're done. Credentials go into macOS Keychain, not a config file.

After that, you right-click a file in Finder, hit "Send to R2," and the public URL lands in your clipboard when it finishes uploading.

Or from the terminal:

r2drop upload screenshot.png
# URL copied to clipboard

r2drop upload ./assets/ --json
# JSON output for scripting

Where rclone wins and it's not close

Sync. If you need to keep a local directory in sync with an R2 bucket (or vice versa), rclone is the tool. rclone sync handles deltas, checksums, and bandwidth limiting. R2Drop doesn't do sync at all.

Mounting. rclone mount myr2:bucket /mnt/r2 gives you R2 as a local filesystem. Useful for certain workflows. R2Drop doesn't do this either.

Cross-platform. rclone runs on everything. R2Drop is macOS only.

Multi-provider. If you're moving files between S3 and R2, or Backblaze and R2, rclone handles that natively. R2Drop only talks to R2.

Where R2Drop wins

The daily workflow of "I have a file, I want it in R2, give me the URL." That loop is 2-3 seconds with R2Drop and probably 15-30 seconds with rclone (open terminal, type the command, construct the URL manually).

Security. Credentials in macOS Keychain vs. a plain text file. Not a huge deal for some people, but it matters to me.

The GUI. Not everyone wants to live in the terminal for every task. The menu bar app lets you drag and drop files, see upload progress, switch accounts, and manage your queue without touching a terminal.

And the CLI shares config with the GUI. You don't configure them separately. r2drop upload uses the same credentials and settings as the menu bar app. With rclone, the config file is the config file, there's no graphical counterpart (unless you count rclone's web GUI, which is... fine).

Can you use both?

Yes. I do. rclone handles my nightly sync jobs. R2Drop handles my ad-hoc uploads during the day. They don't conflict.

If you're already using rclone and happy with it, I'm not going to tell you to switch. But if you find yourself uploading one-off files to R2 regularly and wishing it was faster, give R2Drop a shot.

R2Drop is free, open source, and does one thing well: fast R2 uploads from macOS.
